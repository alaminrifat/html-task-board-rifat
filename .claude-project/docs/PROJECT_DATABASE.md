# Database Schema: html-taskboard

## Tech Stack

- **Database**: PostgreSQL
- **ORM**: TypeORM (NestJS backend)
- **Naming Convention**: snake_case for tables and columns
- **Timestamps**: All tables include `created_at` and `updated_at` (auto-managed by TypeORM `@CreateDateColumn` / `@UpdateDateColumn`)
- **Soft Deletes**: `deleted_at` column where applicable (TypeORM `@DeleteDateColumn`)
- **Primary Keys**: All use `uuid` type (generated by `uuid_generate_v4()`)
- **Timezone**: All timestamps use `timestamptz` (timestamp with time zone)

## 1. Entity List

| # | Entity | Table Name | Description |
|---|--------|------------|-------------|
| 1 | User | `users` | All platform users (Project Owner, Team Member, Admin) |
| 2 | Project | `projects` | Kanban project workspaces |
| 3 | Project Member | `project_members` | Join table: users ↔ projects (many-to-many with role) |
| 4 | Column | `columns` | Board columns representing task statuses |
| 5 | Task | `tasks` | Task cards on the board |
| 6 | Sub-Task | `sub_tasks` | Checklist items within a task |
| 7 | Label | `labels` | Color-coded tags for task categorization (project-scoped) |
| 8 | Task Label | `task_labels` | Join table: tasks ↔ labels (many-to-many) |
| 9 | Comment | `comments` | Threaded comments on tasks |
| 10 | Attachment | `attachments` | File attachments on tasks (stored in AWS S3) |
| 11 | Time Entry | `time_entries` | Time tracking records for tasks |
| 12 | Notification | `notifications` | User notifications |
| 13 | Activity Log | `activity_logs` | Audit trail of all actions |
| 14 | Invitation | `invitations` | Project team member invitations |
| 15 | System Setting | `system_settings` | Admin-configurable system settings (key-value) |
| 16 | Refresh Token | `refresh_tokens` | JWT refresh token storage for session management |
| 17 | Password Reset Token | `password_reset_tokens` | One-time tokens for forgot-password flow |
| 18 | Email Verification Token | `email_verification_tokens` | Tokens for email address verification |
| 19 | User Device | `user_devices` | Registered devices for push notifications (FCM/APNs) |

## 3. Enum Types

### `user_role`

| Value | Description |
|-------|-------------|
| `PROJECT_OWNER` | Creates and manages projects, full board control |
| `TEAM_MEMBER` | Works on assigned tasks, limited permissions |
| `ADMIN` | System-level administrator with full platform access |

### `user_status`

| Value | Description |
|-------|-------------|
| `ACTIVE` | User account is active and can log in |
| `SUSPENDED` | User account is suspended by admin |
| `DELETED` | User account is soft-deleted |

### `project_status`

| Value | Description |
|-------|-------------|
| `ACTIVE` | Project is active and in use |
| `COMPLETED` | Project marked as completed |
| `ARCHIVED` | Project archived by owner or admin |

### `project_role`

| Value | Description |
|-------|-------------|
| `OWNER` | Project owner with full project permissions |
| `MEMBER` | Team member with limited project permissions |

### `task_priority`

| Value | Description |
|-------|-------------|
| `LOW` | Low priority task |
| `MEDIUM` | Medium priority task |
| `HIGH` | High priority task |
| `URGENT` | Urgent priority task |

### `notification_type`

| Value | Description |
|-------|-------------|
| `TASK_ASSIGNED` | User was assigned to a task |
| `DUE_DATE_REMINDER` | Task due date approaching |
| `STATUS_CHANGE` | Task status changed (moved to different column) |
| `COMMENT_MENTION` | User was @mentioned in a comment |
| `NEW_COMMENT` | New comment on a task the user is involved in |
| `INVITATION` | User was invited to a project |

### `invitation_status`

| Value | Description |
|-------|-------------|
| `PENDING` | Invitation sent, awaiting response |
| `ACCEPTED` | Invitation accepted, user joined project |
| `EXPIRED` | Invitation expired (not accepted in time) |
| `CANCELLED` | Invitation cancelled by inviter |

### `board_template`

| Value | Description |
|-------|-------------|
| `DEFAULT` | Standard columns: To Do, In Progress, Review, Done |
| `MINIMAL` | Minimal columns: To Do, Done |
| `CUSTOM` | User-defined custom columns |

### `time_entry_type`

| Value | Description |
|-------|-------------|
| `TIMER` | Recorded via start/stop timer |
| `MANUAL` | Manually entered duration |

### `digest_frequency`

| Value | Description |
|-------|-------------|
| `OFF` | No email digest |
| `DAILY` | Daily email digest |
| `WEEKLY` | Weekly email digest |

### `activity_action`

| `ATTACHMENT_REMOVED` | File attachment removed |
| `MEMBER_ADDED` | Member added to project |
| `MEMBER_REMOVED` | Member removed from project |
| `COLUMN_CREATED` | Column created |
| `COLUMN_UPDATED` | Column updated |
| `COLUMN_DELETED` | Column deleted |
| `PROJECT_UPDATED` | Project settings updated |
| `PROJECT_ARCHIVED` | Project archived |
| `SUB_TASK_ADDED` | Sub-task added |
| `SUB_TASK_COMPLETED` | Sub-task marked complete |

|------------|---------|------|---------|

- Expired tokens are cleaned up by the scheduled token cleanup job

- Raw token is sent to the user via email (registration or email change); only the SHA-256 hash is stored
- Token is single-use: `is_used` is set to `true` after successful verification
- Previous unexpired tokens for the same user are invalidated when a new token is issued (e.g., resend verification)
- Expired tokens are cleaned up by the scheduled token cleanup job
- Used for both initial registration verification and email change re-verification

### 4.19 `user_devices`

| Column | Type | Nullable | Default | Description |
|--------|------|----------|---------|-------------|
| `id` | `uuid` | No | `uuid_generate_v4()` | Primary key |
| `user_id` | `uuid` | No | - | FK → `users.id` (device owner) |
| `token` | `varchar(500)` | No | - | FCM (Android) or APNs (iOS) push notification token |
| `platform` | `varchar(10)` | No | - | Device platform: `ios` or `android` |
| `device_name` | `varchar(100)` | Yes | `NULL` | Human-readable device name (e.g., "iPhone 15 Pro") |

| `deleted_at` | `timestamptz` | Yes | `NULL` | Soft delete timestamp |

**Constraints:**
- `PK_user_devices` PRIMARY KEY (`id`)
- `UQ_user_devices_token` UNIQUE (`token`)
- `FK_user_devices_user` FOREIGN KEY (`user_id`) REFERENCES `users`(`id`) ON DELETE CASCADE

**Indexes:**

| Index Name | Columns | Type | Purpose |
|------------|---------|------|---------|
| `UQ_user_devices_token` | `token` | UNIQUE | Prevent duplicate device registrations |
| `IDX_user_devices_user_id` | `user_id` | BTREE | Lookup devices by user |

**Business Rules:**
- Each user can register multiple devices (phone + tablet, etc.)
- If the same `token` already exists for this user, the registration is idempotent (returns existing device)
- If the same `token` exists for a different user (e.g., shared device), the old registration is replaced
- Push notifications are only sent if `users.push_enabled = true` AND at least one device is registered
- Tokens that repeatedly fail delivery (e.g., FCM returns `NotRegistered`) are automatically removed by a background cleanup job
- Device registrations are deleted when a user logs out from that device
